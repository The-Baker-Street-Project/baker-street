# Build, Release & Installer Redesign

**Date:** 2026-03-10
**Status:** Approved
**Linear:** BAK-54 (cosign/SBOM deferred)

## Problem

The build/release/CI/CD/installer pipeline is unreliable. Every PR risks breaking CI, the installer, or producing wrong-arch images. Root cause: accumulated patches over time instead of a coherent design. The manifest is embedded in the installer binary at compile time, scripts have independent assumptions about versions and secret scoping, and QEMU cross-compilation is slow and flaky.

## Design Principles

1. **The installer is dumb.** Zero embedded knowledge of services, secrets, or manifests. It fetches everything it needs at runtime.
2. **The manifest is generated from reality.** CI produces the manifest from what was actually built, not from a hand-edited file.
3. **Native builds only.** No QEMU. ARM64 images built on ARM64 runners. AMD64 images built on AMD64 runners.
4. **Acceptance test gates every release.** A real install on a kind cluster with real API keys must pass before a release is published.
5. **Dev and install are separate workflows.** `deploy-all.sh` builds from source. The installer pulls from GHCR. Same manifests, different image sources.

## Three Release Artifacts

Every release produces exactly three artifacts attached to the GitHub Release:

| Artifact | Purpose | Size |
|----------|---------|------|
| `manifest.json` | What exists: version, image list with arch availability, URL to install template | ~2KB |
| `install-template.tar.gz` | How to install: config schema (secrets interview) + kustomize-ready K8s manifests | ~50KB |
| `bakerst-install-<os>-<arch>` | The dumb installer binary (2 targets) | ~5MB each |

### Schema Versioning

`manifest.json` and `config-schema.json` each carry a `schemaVersion` (integer, starts at 1). The installer binary knows which schema versions it supports.

- **Supported schema** → proceed
- **Schema too new** → "Your installer is outdated. Download the latest from: \<url\>" and exit
- **Schema missing** → refuse

Manifest and template are always 1:1 (same release, same version). The installer is decoupled but version-checked.

### Config Schema

`config-schema.json` drives the installer's interactive interview:

```json
{
  "schemaVersion": 1,
  "defaults": {
    "namespace": "bakerst",
    "agentName": "Baker"
  },
  "secrets": [
    {
      "key": "ANTHROPIC_API_KEY",
      "description": "Claude API key",
      "inputType": "password",
      "required": false,
      "group": "providers",
      "targetSecrets": ["bakerst-brain-secrets", "bakerst-worker-secrets"]
    },
    {
      "key": "AUTH_TOKEN",
      "description": "API auth token",
      "inputType": "password",
      "required": true,
      "autoGenerate": "hex:32",
      "targetSecrets": ["bakerst-brain-secrets", "bakerst-gateway-secrets"]
    }
  ],
  "features": [
    {
      "key": "telegram",
      "description": "Telegram bot gateway",
      "dependsOn": ["TELEGRAM_BOT_TOKEN"],
      "enables": ["gateway"]
    }
  ],
  "providerValidation": {
    "atLeastOne": ["ANTHROPIC_API_KEY", "OPENAI_API_KEY", "OLLAMA_ENDPOINTS"],
    "message": "At least one AI provider must be configured"
  }
}
```

The installer reads these schemas — it doesn't know what "brain" or "Anthropic" means. It just follows the rules.

## CI/CD Pipeline

Triggered by git tag `v*`. Linear pipeline with a hard gate:

```
Build Images → Build Installer → Package Artifacts → Acceptance Test → Publish Release
```

### Stage 1: Build Images

Per service, 2 parallel jobs:
- **AMD64**: `ubuntu-latest` runner, native Docker build, push to GHCR
- **ARM64**: `ubuntu-24.04-arm` runner, native Docker build, push to GHCR

After both complete: `docker manifest create` to combine into multi-arch manifest.

Services: brain, worker, ui, gateway, sysadmin, voice, nats-sidecar, ext-toolbox, ext-browser, ext-google-workspace (+ any new extensions).

### Stage 2: Build Installer

2 parallel jobs:
- **Linux AMD64**: `ubuntu-latest`, `cargo build --release --target x86_64-unknown-linux-gnu`
- **macOS ARM64**: `macos-latest`, `cargo build --release --target aarch64-apple-darwin`

Generate SHA256 checksums.

### Stage 3: Package Artifacts

- Generate `manifest.json` from actual build outputs (which images built, which arches succeeded)
- Bundle `k8s/` + `config-schema.json` into `install-template.tar.gz`
- The manifest is constructed from reality, not hand-edited

### Stage 4: Acceptance Test (THE GATE)

- Spin up `kind` cluster on GitHub Actions runner
- Load AMD64 images into kind
- Run the just-built Linux installer with test config (real API keys from GitHub Actions secrets)
- Installer applies manifests, creates secrets, waits for rollout
- Verify: pods healthy, endpoints responding, test prompt gets a real response
- **Pass → Stage 5. Fail → pipeline stops. No release.**

### Stage 5: Publish Release

- Create GitHub Release with tag
- Attach: `manifest.json`, `install-template.tar.gz`, installer binaries, checksums
- Send Telegram notification: success or failure at each stage

### Telegram Notifications

The workflow sends Telegram messages at each stage using the existing bot token (stored as GitHub Actions secret):
- "Building 11 images (AMD64 + ARM64)..."
- "Images done. Building installer..."
- "Acceptance test running..."
- "Release v0.6.0 published" or "ACCEPTANCE TEST FAILED — check logs: \<url\>"

## The Dumb Installer

Rust binary with these commands:

```
bakerst-install install [OPTIONS]
bakerst-install status
bakerst-install update
bakerst-install uninstall
```

### Install Options

| Flag | Purpose |
|------|---------|
| `--config <path>` | Skip interview, use saved config (automation/acceptance tests) |
| `--manifest <path>` | Use local manifest file (dev/debugging) |
| `--version <tag>` | Install specific version (default: latest) |
| `--log <path>` | Structured JSON log (default: `./bakerst-install.log`) |
| `--non-interactive` | Fail on missing required values instead of prompting |
| `--dry-run` | Show what would be applied without applying |

### Install Flow

1. **Preflight** — check `kubectl` exists, detect K8s contexts (Docker Desktop, OrbStack, etc.), ask user to pick if multiple
2. **Fetch** — download `manifest.json` from GitHub latest release (or read local `--manifest`)
3. **Schema check** — verify `schemaVersion` is supported
4. **Download template** — fetch and extract `install-template.tar.gz`
5. **Configure** — walk through `config-schema.json` interview (or load `--config`)
6. **Save config** — write to `~/.bakerst/config.json` (reusable for updates)
7. **Pull images** — pull only images needed for selected features, correct arch
8. **Apply** — `kubectl apply -k` extracted manifests with substituted values
9. **Verify** — wait for rollout, check health endpoints, send test prompt if API key provided
10. **Report** — structured JSON log with pass/fail per check, exit 0 or 1

### Other Commands

- **`status`** — reads `~/.bakerst/config.json`, checks pod health, image versions, reports drift
- **`update`** — fetches latest manifest, compares versions, pulls new images, re-applies, re-verifies. Uses saved config, no re-interview.
- **`uninstall`** — `kubectl delete namespace bakerst`, removes `~/.bakerst/`, clean.

## Dev Workflow (`deploy-all.sh` simplified)

Stripped to essentials:

```
deploy-all.sh [--skip-build] [--skip-images] [--dev] [--no-cache] [--version <tag>]
```

### Flow

1. **Preflight** — check docker, kubectl, pnpm, node (≥22 even)
2. **Secrets source** — ask once: "Environment variables (shell) or .env-secrets file?"
3. **Build** — `pnpm install && pnpm -r build` (skip with `--skip-build`)
4. **Docker images** — `scripts/build.sh` with hash-based change detection (skip with `--skip-images`)
5. **Apply** — create K8s secrets, `kubectl apply -k k8s/`
6. **Verify** — same health checks as installer
7. **Summary** — pods, URLs, done

### What's Removed

- 40+ line interactive secret interview (replaced by secrets source prompt)
- Extension-by-extension conditional deployment (kustomize handles it)
- Telemetry deployment (separate: `kubectl apply -k k8s/telemetry/`)
- Blue/green slot management (manifests handle it)
- Per-service rollout timeouts (single verification step)

### Key Difference from Installer

Dev builds local images (`imagePullPolicy: Never`). Installer pulls from GHCR (`imagePullPolicy: IfNotPresent`). A kustomize overlay handles this distinction.

## Target Platforms

### Installer Binaries
- Linux AMD64 (WSL2)
- macOS ARM64 (Apple Silicon)

### Docker Images
- `linux/amd64` (built on `ubuntu-latest`)
- `linux/arm64` (built on `ubuntu-24.04-arm`)

## What This Replaces

All of the following are rewritten from scratch (clean slate):
- `scripts/build.sh`
- `scripts/deploy-all.sh`
- `scripts/deploy.sh`
- `scripts/release.sh`
- `scripts/secrets.sh`
- `.github/workflows/release.yml`
- `tools/installer/` (entire Rust codebase)
- `release-manifest.json` (replaced by CI-generated `manifest.json`)

### What's Preserved
- `.github/workflows/ci.yml` (test/build on PR — works fine)
- `.github/workflows/publish-core.yml` (npm publish — works fine)
- All Dockerfiles (they work, just need minor tweaks for consistency)
- K8s manifests (repackaged into install template, not rewritten)
- `operating_system/` ConfigMap content
